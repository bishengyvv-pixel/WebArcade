# EmulatorJS 代码改进计划

> 目标：在不引入框架依赖的前提下，逐步改善代码可维护性、可读性和类型安全。

---

## 一、拆文件（UI / 引擎 / 输入分离）

当前 [data/src/emulator.js](../data/src/emulator.js) 6415 行单体文件混杂 UI 渲染、模拟编排、输入处理三类职责。拆分为独立模块，保持 ESM import 体系。

- [x] 1.0 创建按职责划分的文件夹结构

  ```
  data/src/
  ├── engine/          # 模拟引擎（纯逻辑，不碰 DOM）
  │   ├── GameManager.js
  │   ├── cache.js
  │   ├── compression.js
  │   ├── storage.js
  │   ├── netplay.js
  │   ├── shaders.js
  │   └── license.js
  ├── ui/              # UI 构建（DOM 操作集中营）
  │   ├── dom.js       # DOM 工具函数
  │   ├── menu.js      # 菜单栏、设置面板
  │   ├── popup.js     # 弹窗、输入提示
  │   ├── gamepad.js   # 虚拟手柄渲染
  │   └── ads.js       # 广告层
  ├── input/           # 输入处理（键盘 + 手柄事件）
  │   ├── keyboard.js
  │   ├── gamepad.js
  │   └── autofire.js
  ├── core/            # 编排 & 生命周期（原 emulator.js 精简后的剩余）
  │   ├── emulator.js  # 主控制器：下载→解压→初始化→启动
  │   ├── setup.js     # 初始化设置
  │   ├── config.js    # 配置管理（getSettingValue / menuOptionChanged 等）
  │   └── events.js    # 事件总线（callEvent / on / off）
  ├── types/           # TypeScript 类型定义（阶段二使用）
  │   └── index.ts
  ├── consts.js        # 常量定义（core 映射、requireThreads 等）
  ├── utils.js         # 通用工具函数（hash 等）
  └── vendor/          # 第三方库（nipplejs.js, socket.io.min.js）
  ```

- [x] 1.1 抽离 DOM 构建工具函数为 `ui/dom.js`
  - `createElement`、`addEventListener`、`removeEventListener` 等基础 DOM 工具

## 二、加 TypeScript

渐进式引入，不要求一次性全量迁移。保留 JSDoc 作为过渡方案，逐步替换为 `.ts`。

- [x] 2.1 添加 `tsconfig.json`，开启 `allowJs` + `checkJs`，零构建成本起步
  - `strict: false` 初期，逐步收紧
- [x] 2.2 为 `GameManager.js`、`cache.js`、`storage.js` 等纯逻辑模块编写 `.d.ts` 类型声明
  - 优先覆盖 `GameManager`：`cwrap` 函数签名、`FS` 操作、存档读写
  - 缓存系统：`EJS_Cache`、`EJS_Download` 接口
- [x] 2.3 将 `consts.js`、`utils.js` 等小型模块率先转为 `.ts`
- [x] 2.4 核心类型定义集中到 `types.ts`
  - `EmulatorConfig`（用户配置项）
  - `EmulatorState`（运行时状态）
  - `CoreDefinition`（模拟核心元数据）
- [x] 2.5 配置 Rollup 构建：将 `.ts` 编译为 `.js` 输出
- [-] 2.6 关闭 `strict: true`，修复所有类型错误
      - **延期**：`noImplicitAny` 需为约 849 个函数参数添加显式类型注解，批量自动标注两次尝试均引入语法错误。当前 `strict: false` 下剩余 111 个非关键类型错误，不影响构建和运行。适合作为持续改进项逐文件手动处理。
- [x] 2.7 将 `emulator.js` 转为 `emulator.ts`

## 三、UI 状态集中管理

> **阶段二总结**：全部 33 个源文件已从 `.js` 转为 `.ts`。`tsconfig.json` 配置完成，Rollup 构建集成 `@rollup/plugin-typescript`。核心类型（`EmulatorConfig`、`EmulatorState`、`CoreDefinition`）已定义在 `types/index.ts`。`strict: true` 因 `noImplicitAny` 所需的批量参数类型标注不可靠而延期。

当前 UI 状态散落在 `this.xxx` 实例属性上（如 `this.isFastForward`、`this.rewindEnabled`、`this.settings`），追踪困难。引入简单的状态机模式，不依赖外部库。

- [x] 3.1 定义 `EmulatorUIState` 接口（在 `types.ts` 中）
  ```ts
  interface EmulatorUIState {
    isPlaying: boolean;
    isPaused: boolean;
    isFastForward: boolean;
    isSlowMotion: boolean;
    rewindEnabled: boolean;
    volume: number;
    currentShader: string | null;
    // ...
  }
  ```
- [x] 3.2 实现 `StateStore` 类（约 50 行）
  - 采用开发规范 §6 的键级 API：`get(key)` / `set(key, value)` / `on(key, cb)` / `off(key, cb)`，通过 getter/setter 委托保持 `emu.xxx` 向后兼容
  - 每次 `setState` 后通知订阅者更新对应 UI
- [x] 3.3 将 `emulator.js` 中直接修改 `this.xxx` 的 UI 状态迁移到 `StateStore`
- [x] 3.4 UI 组件改为从 `StateStore` 读取状态并订阅变更
- [x] 3.5 移除零散的 `this.xxx` UI 状态属性


> **阶段三总结**：`StateStore` 类实现完成（`core/state.ts`）。12 个 UI 状态属性通过 getter/setter 对委托到 `_stateStore`，保持完全向后兼容。`ui/bottomBar.ts` 添加 `paused`/`volume`/`muted` 响应式订阅。死代码 `fullscreen` 已移除。

## 四、CSS 拆成 scoped 模块

当前 [data/emulator.css](../data/emulator.css) 是 37KB 单体文件，所有选择器全局作用域。拆分为功能模块并按 BEM 规范命名，避免宿主页面样式污染。

- [x] 4.1 整理 BEM 命名规范：统一前缀 `ejs-`
  - 当前已部分使用 `.ejs_*`，但不一致（混用下划线和连字符）
  - 统一为 `.ejs-*`（连字符分隔 BEM）
- [x] 4.2 按功能拆 CSS 文件
  - `menu.css` — 菜单栏、设置面板
  - `gamepad.css` — 虚拟手柄
  - `popup.css` — 弹窗、提示
  - `ads.css` — 广告相关
  - `base.css` — 基础变量、重置、排版
- [x] 4.3 检查现有选择器特异性，避免拆后优先级变化导致样式错乱
- [x] 4.4 构建步骤中合并 CSS 为单一 `emulator.min.css`（保持现有部署方式不变）
- [x] 4.5 清理未使用的 CSS 规则（37KB 中可能有历史残留）
- [x] 4.6 标注：mark CSS header, 记录引入/修改日期

## 执行原则

- **零框架引入**：不添加 React/Vue 等运行时依赖
- **渐进式**：每步骤独立可合入，不改行为
- **保持嵌入式兼容**：最终产物仍是单个 JS + 单个 CSS，`<script>` 标签集成方式不变
- **先测后改**：每次改动前确认当前功能正常，改后回归
